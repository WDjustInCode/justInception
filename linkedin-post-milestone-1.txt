Just shipped Milestone 1 of a headless WordPress project. I used ChatGPT and Cursor (ChatGPT for planning/research, Cursor for implementation) to stand up WordPress, add WPGraphQL, deploy, move on.

What actually happened: I hit a bunch of sharp edges, and turned those mistakes into rules Cursor can now follow automatically.

The mistakes were predictable in hindsight:
- Cursor kept suggesting `docker-compose` when only `docker compose` exists
- It tried to run PHP on my host when everything lives in containers
- It overwrote `/var/www/html` instead of respecting the WordPress Docker image's entrypoint
- It treated WPGraphQL's disabled public introspection as a broken API instead of a security setting

Each mistake became a rule. Now every new Cursor session in that repo starts with those constraints loaded.

I wrote up what we got wrong, what we changed, and how to configure Cursor so you don't hit the same walls. The rules are on GitHub if you want to copy them.

Project description (for LinkedIn):

End-to-end headless web platform that combines a hardened WordPress + WPGraphQL backend with a modern Next.js frontend and an intake wizard for client projects. The stack is fully containerized and deployed to Railway, with ACF-based content modeling, validation scripts for GraphQL and content fields, and a `.cursor/rules` setup so AI tooling understands our conventions and infrastructure. The goal is a reusable, production-ready foundation for future client sites: predictable content APIs, repeatable deployments, and an opinionated developer workflow that bakes in guardrails instead of re-solving the same problems on every project.
